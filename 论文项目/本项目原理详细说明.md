## 1. 研究背景与问题定义

本项目目标是构建一套可复现、可扩展、可迁移到实机的 UR10 机械臂综合系统。系统不仅需要完成“机械臂运动规划”，还需满足以下综合要求：

1. 机械结构层面：将 UR10 倒装在三轴龙门架上，并在末端安装铲斗执行器。
2. 仿真执行层面：在 Gazebo Classic 中保证模型可见、控制可执行、状态可回流。
3. 规划层面：通过 MoveIt2 提供稳定规划与执行接口，并与仿真执行链闭环。
4. 感知层面：支持点云采集与障碍物注入规划场景。
5. 实机验证层面：具备独立的 UR 驱动 + MoveIt + 微动验证流程。
6. 交互层面：通过一体化 GUI 将 Real/Sim 两种模式在单页面编排。

问题本质是多子系统一致性问题：

- 模型描述一致性（URDF/Xacro）
- 控制接口一致性（ros2_control + controller manager）
- 状态一致性（/joint_states, TF, planning scene）
- 工具一致性（Gazebo 执行、MoveIt 规划、RViz 观察、GUI 编排）

### 输入-机制-输出-验证

- 输入：任务目标（仿真 + 规划 + 实机 + GUI 一体化）。
- 机制：分层设计（模型、控制、规划、感知、编排）。
- 输出：当前工作区可运行的多主线系统。
- 验证：通过启动链、控制器状态、日志和自动化测试进行交叉验证。

## 2. 系统总体架构

系统由 5 层构成：

1. 模型描述层：`my_robot_xacro/my_robot/urdf/*.xacro`。
2. 仿真执行层：Gazebo + `gazebo_ros2_control` + controller spawner。
3. 规划控制层：MoveIt2（`my_robot_moveit_config` 与历史 `ur10_moveit_config`）。
4. 感知环境层：`ur10_perception`（world、camera、pointcloud、PCD）。
5. 人机编排层：`ur10_unified_gui`（Real/Sim 切换与流程控制）。

其中当前主线是：

- `my_robot` + `my_robot_moveit_config` + `ur10_examples_py` + `ur10_perception`

同时保留历史参考主线：

- `ur10_description` + `ur10_moveit_config` + `ur10_simulation_bringup`

### 数据流与控制流

1. Xacro -> robot_description：
   launch 运行 `xacro` 生成 `robot_description` 参数。
2. robot_description -> Gazebo 与 MoveIt：
   Gazebo 依据描述生成实体；MoveIt 使用同源模型进行规划。
3. MoveIt -> 控制器：
   MoveIt 通过 FollowJointTrajectory 发送轨迹给 `joint_trajectory_controller`。
4. Gazebo -> /joint_states：
   Gazebo 执行后回传状态；MoveIt 与 RViz据此刷新。
5. 感知 -> 规划场景：
   点云/PCD 通过节点注入 collision object。
6. GUI -> 子流程：
   GUI 通过子进程编排 Real/Sim 启动链并解析状态。

### 输入-机制-输出-验证

- 输入：Xacro、launch、controller、MoveIt 配置。
- 机制：参数化启动 + 节点时序编排。
- 输出：可执行的仿真/规划/实机流程。
- 验证：`ros2 launch` 启动完整链路并检查日志与状态收敛。

## 5. my_robot.urdf.xacro 与 my_robot_macro.urdf.xacro 构建机制

> 术语约定：本文将“sarco”明确映射为 `src/my_robot_xacro/my_robot/my_robot.urdf.xacro`，即模型构建入口文件。

### 5.1 `my_robot.urdf.xacro` 的角色

该文件是入口封装层，主要职责：

1. 定义上层参数（如 `ros_hardware_interface`、`inject_gazebo_ros_control_plugin`、`use_gazebo_visual_meshes`）。
2. include `my_robot_macro.urdf.xacro`。
3. 调用宏 `my_robot_model` 完成实体展开。

### 5.2 `my_robot_macro.urdf.xacro` 的角色

该文件是实体定义层，核心机制：

1. 定义 UR10 各 link/joint 的几何、惯量、限制。
2. 定义末端铲斗链接 `shovel_link` 与末端参考 `shovel_tip`。
3. 定义可视网格切换宏 `ur_visual_mesh`：
   - RViz路径可用 DAE
   - Gazebo路径可用 STL（由 `use_gazebo_visual_meshes` 控制）

### 5.3 模型构建算法（工程意义上的“构建算法”）

1. 参数采集：从 launch 传入运行参数。
2. 宏展开：入口文件调用宏文件，展开完整机器人语义树。
3. 网格策略选择：根据参数切换 visual mesh。
4. 结构输出：形成可供 Gazebo、MoveIt、RViz共用的描述模型。

### 输入-机制-输出-验证

- 输入：`my_robot.urdf.xacro` 参数 + macro 文件。
- 机制：xacro 宏展开 + 可视网格条件分支。
- 输出：统一机器人描述。
- 验证：Gazebo 与 RViz显示一致，MoveIt链路可解析模型。

## 7. ros2_control 控制器架构与执行链

核心文件：

- `my_robot_gantry.urdf.xacro`（`ros2_control` system）
- `config/ros2_controllers.yaml`

### 7.1 硬件接口抽象

- 使用 `gazebo_ros2_control/GazeboSystem`。
- 为 UR10 六轴与 gantry 三轴声明 `position` command/state interface。

### 7.2 控制器分层

- `joint_state_broadcaster`：状态广播。
- `joint_trajectory_controller`：UR10 六轴轨迹执行。
- `gantry_trajectory_controller`：门架三轴轨迹执行。

### 7.3 启动时序

`gazebo_gantry.launch.py` 中采用 OnProcessExit 串联：

1. spawn_entity 完成
2. 启动 `joint_state_broadcaster`
3. 启动 `joint_trajectory_controller`
4. 启动 `gantry_trajectory_controller`
5. 运行 `gantry_initializer`

### 输入-机制-输出-验证

- 输入：控制器 YAML + ros2_control 声明。
- 机制：controller_manager spawner 链式启动。
- 输出：可执行的轨迹控制链。
- 验证：`ros2 control list_controllers` 全部 active。

## 14. 工程问题与修复记录

### 14.1 Gazebo 显示与命名兼容

- 问题：中文 link 命名与 mesh 兼容性导致显示/稳定性风险。
- 策略：关键 link 使用稳定命名（如 `shovel_link`），mesh 使用 Gazebo 更稳的加载策略。

### 14.2 mesh 资源策略

- RViz 与 Gazebo 对 visual mesh 兼容性存在差异。
- 通过 `use_gazebo_visual_meshes` 参数切换 DAE/STL，兼顾显示精度与稳定性。

### 14.3 启动时序与残留进程

- 问题：多次启动后旧进程可能污染状态。
- 策略：
  - 主线中使用事件串联 spawner
  - 历史总启动支持 `cleanup_existing_processes` 清理

### 14.4 GUI 可靠性

- 新增互斥策略、防并发冲突。
- 增强停止策略，避免孤儿进程。
- 增强日志解析和状态兜底，避免“无 summary 无结论”。

### 输入-机制-输出-验证

- 输入：故障现象与日志证据。
- 机制：分层排查（模型、控制器、启动时序、进程）。
- 输出：稳定启动与可解释行为。
