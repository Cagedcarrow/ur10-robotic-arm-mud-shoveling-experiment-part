# 本项目节点与调度详解

## 1. 文档目标与阅读方式

本文是《本项目原理详细说明》的节点/调度补充文档，核心回答两件事：

1. 项目里有哪些关键节点（按仿真、实机、GUI 编排分层）。
2. 这些节点如何被 launch 串联，以及如何通过 topic/service/action 互相调动。

建议按三条主线阅读：

- 仿真主线：`my_robot` 与 `ur10_simulation_bringup`
- 实机主线：`ur10_real_comm`
- 编排主线：`ur10_unified_gui`

## 3. 仿真主线调度图

本项目实际存在两条仿真链：

1. `my_robot` 主线：`sim_planning_gantry.launch.py`
2. 历史总链：`complete_simulation.launch.py`

### 3.1 `sim_planning_gantry.launch.py` 调度（当前主线）

#### 节点调度文字图（ASCII）

```text
sim_planning_gantry.launch.py
  ├─ Include: gazebo_gantry.launch.py
  │    ├─ Include: gazebo.launch.py (gzserver/gzclient)
  │    ├─ Node: robot_state_publisher
  │    ├─ Timer(2s) -> Node: spawn_entity.py
  │    ├─ OnProcessExit(spawn_entity)
  │    │    -> Node: spawner joint_state_broadcaster
  │    ├─ OnProcessExit(joint_state_broadcaster)
  │    │    -> Node: spawner joint_trajectory_controller
  │    ├─ OnProcessExit(joint_trajectory_controller)
  │    │    -> Node: spawner gantry_trajectory_controller
  │    └─ OnProcessExit(gantry_trajectory_controller)
  │         -> Node: gantry_control (初始化位置)
  └─ Timer(5s) -> Include: my_robot_moveit.launch.py
       ├─ Node: move_group
       ├─ (可选) Node: rviz2
       └─ (可选) Node: gantry_rviz_control
```

#### 输入 -> 调度机制 -> 输出 -> 验证信号

- 输入：`world`、`gantry_config_file`、`start_rviz`。
- 调度机制：`IncludeLaunchDescription + TimerAction + OnProcessExit`。
- 输出：仿真实体、控制器、MoveIt 规划能力。
- 验证信号：
  - `controller_manager` 控制器 active
  - move_group 日志显示可规划
  - `gantry_control` 返回成功日志

### 3.2 `complete_simulation.launch.py` 调度（历史总编排）

#### 节点调度文字图（ASCII）

```text
complete_simulation.launch.py
  ├─ (可选) ExecuteProcess: cleanup_existing_processes
  ├─ Timer(2.5s) -> Include: gazebo_sim.launch.py
  ├─ Timer(5.0s) -> Include: moveit_planning.launch.py
  ├─ Timer(6.0s) -> (可选) Node: depth_image_viewer
  ├─ Timer(7.0s) -> (可选) Node: gantry_rviz_control
  ├─ Timer(2.5s) -> (可选) Node: synthetic_overhead_camera_node
  ├─ Timer(11.0s) -> (可选) Node: pcd_capture_node
  │    └─ OnProcessExit(pcd_capture) -> Node: pcd_to_collision_scene_node
  │          └─ OnProcessExit(pcd_import) -> Include: demo_nodes.launch.py
  └─ Timer(14.0s, unless import_pcd_obstacle) -> Include: demo_nodes.launch.py
       ├─ (可选) Node: move_group_interface_demo
       └─ (可选) Node: moveit_py_demo
```

#### 输入 -> 调度机制 -> 输出 -> 验证信号

- 输入：`start_cpp_demo/start_py_demo/import_pcd_obstacle/...` 等开关。
- 调度机制：多段 `TimerAction` + 事件触发 `OnProcessExit`。
- 输出：可裁剪的仿真+规划+感知+示例流水线。
- 验证信号：各子节点按条件启动，日志顺序符合调度设计。

## 4. 实机验证主线调度图

入口：`real_comm_moveit_verify.launch.py`。

### 节点调度文字图（ASCII）

```text
real_comm_moveit_verify.launch.py
  ├─ Include: ur_control.launch.py (UR driver)
  ├─ Timer(4s) -> Include: ur_moveit.launch.py (move_group)
  └─ Timer(8s) -> ExecuteProcess: motion_verify_node
       ├─ subscribe: /joint_states
       ├─ publish: /ur10_real_comm/status
       ├─ service: /ur10_real_comm/confirm_execute
       ├─ client: /controller_manager/list_controllers
       ├─ client: /plan_kinematic_path or /move_group/plan_kinematic_path
       └─ action: /<initial_joint_controller>/follow_joint_trajectory
```

### `motion_verify_node` 内部调度状态机

1. 等待 `/joint_states`。
2. 检查控制器服务与 active 状态。
3. 发现 MoveIt 规划服务。
4. 生成 wrist3 微动规划请求。
5. `confirm_execute=true` 时等待确认服务调用。
6. 通过 FollowJointTrajectory action 执行轨迹。
7. 回读关节变化量并发布 PASS/FAIL 状态。

### 输入 -> 调度机制 -> 输出 -> 验证信号

- 输入：`robot_ip/reverse_ip/wrist3_delta_deg/initial_joint_controller/...`。
- 调度机制：`TimerAction` 启动链 + 节点内部状态机。
- 输出：微动验证终态 JSON（status topic + shell summary）。
- 验证信号：`/ur10_real_comm/status` 出现 `"state":"PASS"` 或 `"FAIL"`。

## 5. 感知与障碍物链路调度

### 5.1 关键节点职责

- `synthetic_overhead_camera_node`：定时发布合成点云（PointCloud2）。
- `pcd_capture_node`：订阅点云，降采样后写 PCD 并退出。
- `pcd_to_collision_scene_node`：加载 PCD，提取障碍包围盒，写入 MoveIt planning scene。

### 5.2 调度关系

```text
synthetic_overhead_camera_node -> pointcloud_topic
      ↓
pcd_capture_node (保存 pcd)
      ↓ OnProcessExit
pcd_to_collision_scene_node (注入 collision object)
      ↓ OnProcessExit
move_group_interface_demo / moveit_py_demo（在障碍场景下规划）
```

### 输入 -> 调度机制 -> 输出 -> 验证信号

- 输入：`pointcloud_topic`、`pcd_file`、`obstacle_id`。
- 调度机制：topic 数据流 + `OnProcessExit` 串联。
- 输出：规划场景内障碍对象。
- 验证信号：`pcd_to_collision_scene_node` 日志包含 injected table/box 信息。

## 6. GUI 编排层如何驱动节点链

入口：`ur10_unified_gui/main.py`。

### 6.1 两种模式

- Real 模式：调用 `run_full_verify.sh`（主）或 `run_full_verify_wsl.sh`（回退）。
- Sim 模式：调用 `ros2 launch ur10_simulation_bringup complete_simulation.launch.py ...`。

### 6.2 互斥与停止策略

- 互斥：`_guard_mutex()`，当前有运行态会话时拒绝启动另一模式。
- 停止：`ProcessSession.stop()` 使用进程组 `SIGTERM -> SIGKILL`。

### 6.3 状态汇总策略

- 优先解析日志 `[SUMMARY] state=... reason=...`。
- 若无 summary，按退出码兜底：`exit=0 -> PASS`，其他 -> FAIL。

### 输入 -> 调度机制 -> 输出 -> 验证信号

- 输入：GUI 表单参数 + 按钮事件。
- 调度机制：subprocess + log queue + 状态机。
- 输出：统一状态栏和可导出日志。
- 验证信号：状态栏收敛到 PASS/FAIL/WARN/STOPPED。

## 8. 时序级说明（触发机制细化）

### 8.1 IncludeLaunchDescription
用于拼接子 launch，构成多层启动树：
- 仿真链：主 launch -> gazebo/moveit/demo 子 launch
- 实机链：real_comm launch -> driver + moveit

### 8.2 TimerAction
用于显式延时，避免“先起的服务未就绪导致后续节点失败”：
- `sim_planning_gantry` 中 MoveIt 延时 5 秒
- `complete_simulation` 中 gazebo/moveit/demo 分段延时
- `real_comm` 中 driver(0s)、moveit(4s)、verify(8s)

### 8.3 OnProcessExit
用于“前一阶段完成后再进入下一阶段”：
- spawn 完后再依次起三个控制器
- pcd_capture 结束后再 pcd_import
- pcd_import 结束后再 demo

### 输入 -> 调度机制 -> 输出 -> 验证信号

- 输入：launch 参数与阶段依赖。
- 调度机制：Include + Timer + OnProcessExit 组合。
- 输出：稳定、可预测的启动顺序。
- 验证信号：日志顺序与依赖关系一致。

## 9. 常见故障定位（按调度断点）

### 9.1 节点没起

- 先看 launch 条件参数是否关闭了节点（如 `start_py_demo=false`）。
- 再看是否被上游阶段卡住（例如等待 `OnProcessExit` 触发）。

### 9.2 接口没连上

- action 不可用：通常控制器未 active 或 controller 名不匹配。
- service 不可用：通常 move_group 尚未 ready 或命名空间不一致。
- topic 无数据：通常上游传感器/状态广播节点没启动。

### 9.3 调度中断

- `TimerAction` 时间过短可能导致后续节点“抢跑”。
- `cleanup_existing_processes` 关闭时，旧残留进程可能污染新会话。
- GUI 中如果互斥命中，第二条链会被故意拒绝启动（不是故障）。

### 快速排查顺序

1. 看 launch 顶层日志是否进入对应阶段。
2. 看关键节点是否创建成功（move_group、controller spawner、verify node）。
3. 看关键接口是否可见（joint_states、plan service、follow_joint_trajectory）。
4. 看状态总结（`/ur10_real_comm/status` 或 GUI `[SUMMARY]`）。

## 10. 论文可引用版本（术语统一与图表建议）

- “调度”：launch 级触发机制（Include/Timer/OnProcessExit）。
- “交互接口”：topic/service/action。
- “执行闭环”：规划请求 -> 控制器执行 -> joint state 回流 -> 状态判断。

### 10.2 图表建议

### 10.3 可直接引用的工程结论

1. 本项目通过 `IncludeLaunchDescription + TimerAction + OnProcessExit` 组合实现了稳定可控的多阶段调度。
2. 节点间调用采用 ROS2 标准三接口（topic/service/action）分工：状态流、请求流、执行流清晰分离。
3. GUI 作为编排层不侵入核心算法，仅承担流程驱动、状态汇总与故障可视化职责。
