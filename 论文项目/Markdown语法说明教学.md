## 1. 文档用途与适用范围

### 写法
使用一级标题定义文档主题，先放目录，再按章节组织内容。

### 示例
本文件就是示例：标题 + 目录 + 分章节。

### 常见坑
- 直接开写，不做目录，后期章节一多难维护。
- 一级/二级标题混乱，导致结构不可读。

### 建议用法
- 在论文项目中，优先使用“一级标题 1 个 + 二级标题多节 + 三级标题讲细项”的结构。
- 每节使用固定小模板：`写法 / 示例 / 常见坑 / 建议用法`。

## 2. 标题层级规范

### 写法

```markdown
# 一级标题（整篇文档仅 1 个）
## 二级标题（章节）
### 三级标题（章节内主题）
#### 四级标题（非常细的点）
```

### 示例

```markdown
## 4. 列表语法
### 4.1 有序列表
#### 4.1.1 适合步骤描述
```

### 常见坑
- 跳级：`##` 后直接 `####`。
- 为了看起来大字，乱用多个 `#`。
- 标题中包含过多标点，影响锚点可读性。

### 建议用法
- 最多用到 `####`。
- 标题尽量短而准，避免“一行长句”。

## 3. 段落、换行、分隔线、强调

### 写法

```markdown
普通段落直接换行。

空一行表示新段落。

行尾加两个空格  
可实现同段内换行。

---

**粗体**、*斜体*、`行内代码`
```

### 示例
你可以把关键术语写成 `robot_description`、`move_group`、`OnProcessExit`。  
同段内换行可用于“定义 + 注释”。

---

### 常见坑
- 用很多空格硬凑排版。
- 行内代码和代码块混用，导致阅读混乱。

### 建议用法
- 术语/命令/路径统一使用反引号。
- 强调用粗体，不要整段加粗。

---

## 4. 列表语法

### 写法

```markdown
- 无序项 A
- 无序项 B

1. 有序步骤 1
2. 有序步骤 2
3. 有序步骤 3

- [ ] 任务未完成
- [x] 任务已完成
```

### 示例

1. 启动仿真主线。  
2. 检查控制器状态。  
3. 记录日志与结果。  

- [x] 已完成节点调度文档
- [ ] 待补论文图表编号

### 常见坑
- 有序列表手写 `1) 2)`，在 Markdown 中兼容性差。
- 列表内换行不缩进，导致格式错乱。

### 建议用法
- 步骤类内容统一用有序列表。
- 清单类内容统一用任务列表。

## 5. 代码块语法

### 写法

```markdown
```bash
ros2 launch my_robot sim_planning_gantry.launch.py
```

```python
def main():
    print("hello")
```

```text
A -> B -> C
```
```

### 示例

```bash
source /opt/ros/humble/setup.bash
source /root/moveit_ws/install/setup.bash
source /root/ur10_ws/install/setup.bash
```

```text
spawn_entity -> joint_state_broadcaster -> joint_trajectory_controller -> gantry_trajectory_controller
```

### 常见坑
- 代码块开头三个反引号，结尾忘了闭合。
- 未写语言标记（`bash/python/text`），可读性下降。

### 建议用法
- 命令类统一 `bash`。
- 流程图类文字统一 `text`。

## 6. 表格语法

### 写法

```markdown
| 列1 | 列2 | 列3 |
| --- | --- | --- |
| A | B | C |
```

### 示例（复用本项目节点表风格）

| 节点名 | 包名 | 关键接口 |
| --- | --- | --- |
| `move_group` | `moveit_ros_move_group` | MotionPlanService |
| `gantry_control` | `ur10_examples_py` | FollowJointTrajectory action |
| `motion_verify_node` | `ur10_real_comm` | `/ur10_real_comm/status` |

### 常见坑
- 分隔行少写 `---`，表格不渲染。
- 列数不一致（某行多/少一个 `|`）。

### 建议用法
- 论文里“节点清单、接口映射、参数对照”都优先用表格。

## 7. 链接与锚点

### 写法

```markdown
[显示文本](相对路径或URL)
[跳转到第3章](#3-段落换行分隔线强调)
```

### 示例

- [本项目原理详细说明](./本项目原理详细说明.md)
- [本项目节点与调度详解](./本项目节点与调度详解.md)
- [跳到“常见错误与排查”](#10-常见错误与排查)

### 常见坑
- 中文标题锚点手写错误。
- 路径写成绝对路径，换机器后失效。

### 建议用法
- 目录中的锚点建议由编辑器自动生成/复制。
- 项目内文档尽量用相对路径链接。

## 8. 图片与图注

### 写法

```markdown
![图注文本](./images/figure_01_arch.png)

图 1-1 系统总体架构图。
```

### 示例

```markdown
![仿真主线调度图](./images/fig_sim_pipeline.png)

图 3-1 仿真主线节点调度图（由 launch 时序抽象）。
```

### 常见坑
- 图片只写本地绝对路径，GitHub 无法显示。
- 图注和图片分离太远。

### 建议用法
- 建议命名：`fig_章节号_主题.png`，例如 `fig_04_real_verify_flow.png`。
- 图片后紧跟图注，便于论文引用。

## 9. 引用块与提示块

### 写法

```markdown
> 这是引用块。
> 可用于说明、定义或注意事项。
```

### 示例

> 注意：`EXIT_CODE=124` 在 `timeout` 场景通常表示“超时主动终止”，不等于流程失败。

> 结论：验证“改动生效”必须包含 A/B 对照，而不仅是“系统能启动”。

### 常见坑
- 引用块写成长段正文，导致主线阅读被打断。

### 建议用法
- 引用块用于“结论、注意、定义”，正文保留流程叙述。

## 10. 常见错误与排查

### 写法
用“问题 -> 原因 -> 修复”的三段格式。

### 示例

1. 问题：目录点击无跳转。  
   原因：锚点拼写不一致。  
   修复：复制编辑器生成的锚点，不手敲。  

2. 问题：代码块后面整篇都变成代码样式。  
   原因：代码块未闭合。  
   修复：补齐结尾 ```。  

3. 问题：表格显示错位。  
   原因：某一行列数不一致。  
   修复：保证每行 `|` 数一致。  

### 建议用法
- 每写完一节，先在 VSCode 预览（`Ctrl+Shift+V`）检查渲染。

## 11. 论文写作推荐模板

### 写法（章节骨架）

```markdown
## X. 章节标题

### X.1 背景与目标

### X.2 方法与机制

### X.3 输入 -> 机制 -> 输出 -> 验证

### X.4 结果与讨论
```

### 示例（节点调度章节）

```markdown
## 4. 实机验证主线调度图

### 4.1 调度入口
入口为 `real_comm_moveit_verify.launch.py`。

### 4.2 ASCII 时序图
```text
driver -> move_group -> motion_verify_node
```

### 4.3 输入 -> 机制 -> 输出 -> 验证
- 输入：robot_ip, reverse_ip
- 机制：TimerAction 分段启动
- 输出：状态消息
- 验证：/ur10_real_comm/status
```

### 常见坑
- 只写“现象”，不写“机制”。
- 章节之间重复粘贴大量相同段落。

### 建议用法
- 每章至少给一个“可验证信号”。
- 方法章节优先写机制链，而不是堆形容词。

## 12. 与本项目文档风格对齐规范

### 写法
统一术语、编号和表格字段。

### 示例规范

- 术语统一：
  - “调度机制”仅指 `IncludeLaunchDescription / TimerAction / OnProcessExit`
  - “接口”仅指 `topic/service/action`
- 表格字段推荐顺序：
  - 节点类：`节点名 | 包名 | 启动入口 | 角色 | 关键 I/O`
  - 接口类：`接口名 | 调用方 | 提供方 | 用途`

### 常见坑
- 同一概念多种叫法（如“流程/链路/调度”混用无定义）。
- 章节编号格式不统一。

### 建议用法
- 先定义术语再写正文。
- 全文统一使用“输入 -> 机制 -> 输出 -> 验证”小节模板。

## 结语

这份教程的目标不是“记住所有 Markdown 语法”，而是让你快速写出结构稳定、术语统一、可直接进入论文素材库的工程文档。你可以从附录模板开始，每次只替换章节内容，不改骨架。
