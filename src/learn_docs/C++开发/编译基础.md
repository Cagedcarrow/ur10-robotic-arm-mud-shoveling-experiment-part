# ROS 2 编译基础

## 一句话理解

编译就是把源码转换成系统能直接运行或加载的结果。对 ROS 2 来说，编译不仅是生成可执行文件，还包括安装 launch、配置、消息接口和 Python 入口。

## 1. 什么是编译

可以把编译理解成“把源码整理成可运行状态”的过程。

### 对 C++ 包

通常会发生：

- `.cpp` -> 目标文件 -> 可执行文件 / 库

### 对 Python 包

虽然 Python 不像 C++ 那样编译成本地机器码，但在 ROS 2 工作流里，`colcon build` 仍然很重要，因为它会：

- 安装 Python 包
- 建立 `ros2 run` 的命令入口
- 安装 launch / config / rviz / meshes 等资源
- 刷新工作空间中的安装结果

## 2. ROS 2 中的构建流程

一个典型过程：

```text
src/ -> build/ -> install/
```

含义是：

- `src/`：源码
- `build/`：构建中间过程
- `install/`：最终被运行时使用的结果

## 3. 最常用的构建命令

### 构建整个工作空间

```bash
colcon build
```

### 只构建指定包

```bash
colcon build --packages-select my_package
```

### Python 开发推荐的方式

```bash
colcon build --packages-select my_package --symlink-install
```

### 构建后加载环境

```bash
source install/setup.bash
```

## 4. 为什么 `--symlink-install` 很重要

对 Python 包来说，`--symlink-install` 很常用。

作用是：

- 不把 Python 文件硬复制到 `install/`
- 而是建立符号链接

这样做的好处是：

- 修改 Python 源码后，很多情况下不用重新复制文件
- 开发迭代更快

### 推荐习惯

如果主要写 Python 节点，尽量长期使用：

```bash
colcon build --symlink-install
```

## 5. 什么时候必须重新编译

### 一定要重新编译的情况

- 修改了 `CMakeLists.txt`
- 修改了 `setup.py`
- 修改了 `package.xml` 并影响构建或安装
- 修改了 `.cpp` / `.hpp`
- 修改了 `.msg` / `.srv` / `.action`
- 新增了 launch、config、rviz、mesh 等需要安装的文件，但安装规则之前没覆盖到

### 可能需要重新编译的情况

- 修改 Python 文件，但没有使用 `--symlink-install`
- 修改 launch / config 文件，但安装目录里还是旧内容

### 通常不需要重新编译的情况

- 使用 `--symlink-install` 后，只改 Python 源码
- 只是查看文档、改注释

## 6. 一个实用判断表

| 修改内容 | 是否通常需要重新编译 |
| --- | --- |
| `.cpp` / `.hpp` | 需要 |
| `.msg` / `.srv` / `.action` | 需要 |
| `setup.py` | 需要 |
| `CMakeLists.txt` | 需要 |
| `package.xml` | 通常需要 |
| Python `.py`（配合 `--symlink-install`） | 通常不需要 |
| Python `.py`（未使用 `--symlink-install`） | 通常需要 |
| 文档 `.md` | 不需要 |

## 7. 常见命令组合

### 最稳的单包构建流程

```bash
cd ~/ros2_ws
source /opt/ros/humble/setup.bash
colcon build --packages-select my_package --symlink-install
source install/setup.bash
```

### 构建后直接运行

```bash
ros2 run my_package my_node
```

### 构建后启动 launch

```bash
ros2 launch my_package demo.launch.py
```

## 8. 常见错误

### 修改完代码运行结果还是旧的

常见原因：

- 没编译
- 没重新 source
- Python 包没用 `--symlink-install`

### launch 找不到文件

常见原因：

- launch 没有被安装进 `install/`
- 构建后没有重新 source

### 修改接口后下游包报错

常见原因：

- 修改了消息或服务定义，但依赖它的包没有重新构建

## 9. 清理重编的做法

如果工作区状态混乱，可以直接清理后重建：

```bash
rm -rf build/ install/ log/
colcon build --symlink-install
```

这是最慢但最稳的恢复方式。

## 10. 推荐习惯

### Python 项目

- 优先使用 `--symlink-install`
- 改完关键配置仍然重新 build 一次

### C++ 项目

- 改动后及时构建对应包
- 不确定时优先单包重建

### 接口包

- 改了 `msg/srv/action` 后，不只重建自己，还要关注依赖它的包

## 11. 快速记忆

```text
改源码，先想它会不会影响 install 结果
会影响，就重编
不确定，就单包重编
工作区乱了，就清理后重编
```
