# 本次新增与修改文件说明

## 这篇文档解决什么问题

这篇文档专门回答两个问题：

- 这次到底创建了哪些文件、改了哪些文件。
- 每个文件存在的意义是什么，它在整套系统里扮演什么角色。

对于 ROS 2 新手来说，理解“文件和文件之间是什么关系”，往往比一开始就理解全部原理更重要。

## 先建立一个总认识

这次项目不是只改了一个 URDF 文件，而是一起改了下面几层东西：

- 机器人描述层
- Gazebo 仿真层
- 控制器配置层
- MoveIt 规划层
- Python 示例与控制脚本层
- 文档层

你可以把这些文件想象成一条流水线：

Xacro 定义机器人结构，launch 把它们启动起来，controller 配置让它们能被控制，MoveIt 配置让它们能被规划，Python 脚本让你能直接发动作命令。

## 核心文件总表

下面这些是这次最关键的文件或目录。

### 1. `my_robot/urdf/my_robot_macro.urdf.xacro`

作用：

- 把 `my_robot` 机械臂本体封装成一个可复用的 Xacro 宏。
- 定义 UR10 各个 link、joint、末端铲斗、`shovel_tip` 和 `ros2_control` 接口。

它解决了什么问题：

- 如果不先把机械臂本体封装成宏，就很难把它优雅地挂到龙门架上。
- 宏化之后，机械臂就不再只能“独立存在”，而是可以被别的外层结构复用。

它被谁使用：

- `my_robot.urdf.xacro`
- `my_robot_gantry.urdf.xacro`

如果改坏会出现什么现象：

- 机械臂 link/joint 丢失。
- MoveIt 规划组无法匹配关节。
- Gazebo 中机械臂显示异常或控制器加载异常。

### 2. `my_robot/my_robot.urdf.xacro`

作用：

- 这是机械臂本体的入口文件。
- 它主要负责把参数传给 `my_robot_macro.urdf.xacro`。

它解决了什么问题：

- 给“只看机械臂本体”的场景保留一个简单入口。
- 避免每次都直接操作大而复杂的宏定义文件。

它被谁使用：

- 独立查看机械臂模型的场景。
- 某些只需要本体描述的工具链。

如果改坏会出现什么现象：

- 单独查看 `my_robot` 本体时参数不生效。
- 宏参数无法传递到机器人定义内部。

### 3. `my_robot/urdf/my_robot_gantry.urdf.xacro`

作用：

- 这是“龙门架 + 机械臂”组合模型的核心文件。
- 定义 world、龙门架固定结构、X/Y/Z 三个滑台、安装板，以及机械臂挂载关系。

它解决了什么问题：

- 让机械臂不再固定在世界原点，而是能通过龙门架三轴改变基座位置。

它被谁使用：

- Gazebo 仿真启动
- MoveIt 启动
- `robot_state_publisher`

这次为什么重要：

- 这是整个组合机器人的真正源头。
- 只要这份描述正确，Gazebo、RViz2、MoveIt、TF 才能看到同一个机器人。

如果改坏会出现什么现象：

- 坐标树断裂。
- 龙门架关节不存在。
- 机械臂安装位置错误。
- Gazebo 或 MoveIt 使用的机器人结构不一致。

### 4. `my_robot/config/ros2_controllers.yaml`

作用：

- 定义 `controller_manager` 要加载哪些控制器。
- 定义每个控制器负责哪些 joints。

本文件中的两个主控制器：

- `joint_trajectory_controller`
  - 负责 UR10 机械臂 6 个关节。
- `gantry_trajectory_controller`
  - 负责龙门架的 `gantry_x_joint`、`gantry_y_joint`、`gantry_z_joint`。

它解决了什么问题：

- 把“机械臂控制”和“龙门架控制”从逻辑上拆开。
- 让 MoveIt 只管机械臂轨迹，龙门架由专门控制器管理。

如果改坏会出现什么现象：

- 控制器加载失败。
- 轨迹发送到了错误的 joints。
- MoveIt 执行时找不到匹配控制器。

### 5. `my_robot/config/gantry_config.json`

作用：

- 保存龙门架初始位置、运动范围和结构高度。

目前包含的关键字段：

- `initial.x/y/z`
- `limits.x.min/max`
- `limits.y.min/max`
- `limits.z.min/max`
- `structure.base_height`

它解决了什么问题：

- 把“结构参数”和“机器人描述模板”分开。
- 让你不必每次修改 Xacro 才能改龙门架范围或初始位置。

它被谁使用：

- `gazebo_gantry.launch.py`
- `sim_planning_gantry.launch.py`
- `my_robot_moveit.launch.py`

如果改坏会出现什么现象：

- 启动时读取配置失败。
- 龙门架限位不合理。
- 初始位置不对，甚至把机械臂放到不可见或不安全的位置。

### 6. `my_robot/launch/gazebo_gantry.launch.py`

作用：

- 负责 Gazebo 这一侧的完整启动链路。

主要做的事情：

- 读取 `gantry_config.json`
- 调用 xacro 生成 `robot_description`
- 启动 Gazebo
- 启动 `robot_state_publisher`
- 调用 `spawn_entity.py` 把模型放进 Gazebo
- 顺序加载控制器
- 最后调用龙门架初始化动作

它解决了什么问题：

- 把“生成模型、进仿真、加载控制器、初始化龙门架”连成一条稳定启动链路。

如果改坏会出现什么现象：

- Gazebo 没有模型。
- 控制器没有被加载。
- 龙门架初始化不执行。

### 7. `my_robot/launch/sim_planning_gantry.launch.py`

作用：

- 这是你平时最常用的总入口启动文件。

它主要把两条链路拼到一起：

- Gazebo 仿真链路
- MoveIt + RViz2 规划链路

它解决了什么问题：

- 让你只用一个命令，就同时得到仿真和规划。

如果改坏会出现什么现象：

- Gazebo 和 MoveIt 其中一侧起不来。
- 两边虽然都起来了，但使用的模型参数不一致。

### 8. `my_robot_moveit_config/`

作用：

- 这是为新组合机器人专门新建的 MoveIt 配置包。

这个包存在的意义非常大：

- 之前的 MoveIt 配置不一定适合“龙门架 + 新末端定义”的机器人。
- 所以需要新的 SRDF、运动学参数、控制器映射和 RViz2 配置。

目录内关键文件如下。

#### `srdf/my_robot.srdf.xacro`

作用：

- 定义规划组、末端执行器、禁碰、自碰撞规则等语义信息。

为什么重要：

- URDF 只描述“物理结构”。
- SRDF 描述“规划语义”。

#### `config/kinematics.yaml`

作用：

- 定义逆运动学求解相关配置。

#### `config/joint_limits.yaml`

作用：

- 给 MoveIt 使用的关节限制参数。

#### `config/ompl_planning.yaml`

作用：

- 配置 OMPL 规划器参数。

#### `config/moveit_controllers.yaml`

作用：

- 告诉 MoveIt：执行轨迹时应该把请求发给哪个控制器。

#### `launch/my_robot_moveit.launch.py`

作用：

- 启动 `move_group` 和 RViz2。
- 同时构造与 Gazebo 一致的 `robot_description` 和 `robot_description_semantic`。

#### `rviz/view_robot.rviz`

作用：

- 定义 RViz2 打开后的显示布局。
- 包含 MotionPlanning 面板和龙门架交互 Marker 显示项。

### 9. `ur10_examples_py/ur10_examples_py/moveit_py_demo.py`

作用：

- 这是通用化后的 MoveIt Python 示例程序。

它解决了什么问题：

- 不再只服务某一个固定机器人。
- 现在可以复用到 `my_robot` 项目里执行规划与轨迹下发。

为什么重要：

- 它是验证“MoveIt 规划成功并最终在 Gazebo 中执行成功”的关键测试入口。

如果改坏会出现什么现象：

- 规划脚本无法连接 `move_group`。
- 轨迹无法发送给控制器。
- 不能作为自动化验证工具使用。

### 10. `ur10_examples_py/ur10_examples_py/gantry_control.py`

作用：

- 通过 `FollowJointTrajectory` action 给 `gantry_trajectory_controller` 发送龙门架三轴位置目标。

它解决了什么问题：

- 让龙门架可以通过脚本方式移动，而不是只能手工改配置。

### 11. `ur10_examples_py/ur10_examples_py/gantry_rviz_control.py`

作用：

- 在 RViz2 里创建 Interactive Marker。
- 通过拖拽 Marker 来控制龙门架的 X/Y/Z 位置。

它解决了什么问题：

- 给用户一个更直观的交互方式。
- 让“移动龙门架，从而改变机械臂基座位置”这件事变得可视化。

## 当前 Gazebo 与 RViz2 的模型策略

当前最终策略是：

- Gazebo 使用从原始 DAE 转换出来的高保真 STL visual mesh。
- RViz2 继续使用原始 DAE mesh。

也就是说，两边看到的外形都应当是真实机械臂外形，但资源格式按渲染器分开：

- RViz2 使用：

- `base.dae`
- `shoulder.dae`
- `upperarm.dae`
- `forearm.dae`
- `wrist1.dae`
- `wrist2.dae`
- `wrist3.dae`

- Gazebo 使用：

- `base_gazebo_visual.stl`
- `shoulder_gazebo_visual.stl`
- `upperarm_gazebo_visual.stl`
- `forearm_gazebo_visual.stl`
- `wrist1_gazebo_visual.stl`
- `wrist2_gazebo_visual.stl`
- `wrist3_gazebo_visual.stl`

这套策略的重点是精度优先，同时绕开 Gazebo Classic 对原始 COLLADA visual 的兼容性问题。

## 如果你只记住一件事

这次的文件改动虽然多，但最核心的逻辑其实很简单：

先用 Xacro 正确描述组合机器人，再让 Gazebo、MoveIt、RViz2、控制器都围绕同一份描述工作。

只要这一点成立，整个系统就稳定。
