# 龙门架是怎么构建的

## 这篇文档解决什么问题

这篇文档专门解释：

- 龙门架在 URDF/Xacro 里是怎么一层层搭出来的。
- 机械臂为什么能“挂”到龙门架上。
- 龙门架三轴为什么能够改变机械臂基座的位置。

这篇文档的目标不是只告诉你“文件名是什么”，而是让你看懂它的结构逻辑。

## 先理解一个基本概念

在 URDF/Xacro 里，机器人是这样描述的：

- `link` 表示一个刚体部件。
- `joint` 表示两个部件之间的连接关系。

如果把它画成树：

- 每个 `link` 是一个节点。
- 每个 `joint` 是节点之间的边。

所以，“构建一个龙门架”本质上就是：

定义若干个 `link`，再用若干个 `joint` 把它们按正确的顺序连起来。

## 本项目里龙门架的总体结构

本项目组合机器人可以概括成下面这条链：

```text
world
  -> gantry_structure_link
    -> gantry_x_carriage_link
      -> gantry_y_carriage_link
        -> gantry_z_carriage_link
          -> ur_mount_link
            -> ur10
              -> ... 机械臂其他关节 ...
```

你可以这样理解：

- `world` 是世界坐标原点。
- `gantry_structure_link` 是龙门架固定框架。
- `gantry_x_carriage_link` 是 X 向移动滑台。
- `gantry_y_carriage_link` 是 Y 向移动滑台。
- `gantry_z_carriage_link` 是 Z 向移动滑台。
- `ur_mount_link` 是机械臂安装板。
- `ur10` 是机械臂基座 link。

## 第 1 层：世界坐标 `world`

在 `my_robot_gantry.urdf.xacro` 中，先定义了：

```xml
<link name="world" />
```

它的意义不是“一个真实零件”，而是整个机器人系统挂接到世界中的参考坐标。

为什么必须有它：

- Gazebo 需要知道机器人相对于世界的位置。
- RViz2 和 TF 也需要一个稳定的全局参考。

## 第 2 层：龙门架固定结构 `gantry_structure_link`

这个 link 代表龙门架本身不动的那部分框架：

- 左立柱
- 右立柱
- 顶梁

这些 visual 是用 `box` 拼出来的，目的不是做工业级精细建模，而是清楚表达龙门架的空间结构。

它通过一个固定关节连到世界：

```xml
<joint name="world_to_gantry_structure" type="fixed">
```

这说明：

- 龙门架整体框架在世界里是固定的。
- 会动的不是框架本身，而是后面的滑台。

## 第 3 层：X 轴滑台

对应的 link 和 joint 是：

- `gantry_x_carriage_link`
- `gantry_x_joint`

这个 joint 是 `prismatic`，也就是平移关节，不是旋转关节。

```xml
<joint name="gantry_x_joint" type="prismatic">
```

为什么用 `prismatic`：

- 因为龙门架在 X 方向上是“直线移动”，不是“转动”。

关键参数：

- `axis xyz="1 0 0"`
  - 表示沿 X 方向移动。
- `limit lower/upper`
  - 定义 X 轴最小和最大行程。

## 第 4 层：Y 轴滑台

对应的 link 和 joint 是：

- `gantry_y_carriage_link`
- `gantry_y_joint`

它被挂在 X 轴滑台下面，所以它的运动是“在 X 轴当前所在位置的基础上，再沿 Y 方向移动”。

关键参数：

- `axis xyz="0 1 0"`
  - 表示沿 Y 方向平移。

为什么这样设计：

- 这符合“串联机构”的思路。
- 先有 X 位置，再在这个位置上做 Y 偏移。

## 第 5 层：Z 轴滑台

对应的 link 和 joint 是：

- `gantry_z_carriage_link`
- `gantry_z_joint`

关键参数：

- `axis xyz="0 0 1"`
  - 表示沿 Z 方向上下移动。

为什么它很重要：

- 机械臂最终挂在它下面。
- 所以 Z 轴一动，机械臂基座的高度就跟着变。

## 第 6 层：机械臂安装板 `ur_mount_link`

这个 link 是龙门架和机械臂之间的过渡层。

它的存在意义是：

- 给机械臂一个明确的安装面。
- 不把机械臂直接硬连到 Z 轴滑台本体上。
- 以后如果你要改安装偏移、法兰板尺寸、安装姿态，有一个单独层可以调整。

对应固定关节：

```xml
<joint name="gantry_to_ur_mount" type="fixed">
```

这说明：

- 安装板相对 Z 轴滑台是固定的。
- 它本身不单独运动。

## 第 7 层：把机械臂挂到安装板上

关键关节是：

```xml
<joint name="ur_mount_to_my_robot" type="fixed">
```

它做的事情是：

- 父 link 是 `ur_mount_link`
- 子 link 是 `ur10`

也就是说：

龙门架真正挂的不是“整台机械臂这个抽象概念”，而是机械臂基座 link `ur10`。

这一步非常关键，因为它直接决定了：

- 机械臂基座在世界中的位置
- 机械臂基座的朝向
- 机械臂后续所有 link 的坐标基准

## 为什么 `my_robot` 要先拆成宏

如果不拆宏，你很难把机械臂优雅地嵌入新的外部结构。

拆成宏之后，好处有三个：

1. 机械臂本体可以独立复用。
2. 龙门架外层可以在合适的位置调用这段宏。
3. 同一份机械臂定义可以服务不同入口文件，而不需要复制粘贴整台机器人。

这正是 `my_robot_macro.urdf.xacro` 的价值。

## `gantry_config.json` 和 `my_robot_gantry.urdf.xacro` 的关系

这两者的关系是：

- `my_robot_gantry.urdf.xacro` 是模板。
- `gantry_config.json` 是模板的参数来源。

JSON 里给出的关键值会在 launch 阶段被读出来，再传给 xacro：

- `initial.x/y/z`
- `limits.x/y/z`
- `structure.base_height`

也就是说：

Xacro 决定“结构长什么样”，JSON 决定“这次实例化时用什么参数”。

## 这些参数分别是什么意思

### `gantry_base_height`

物理意义：

- 龙门架顶梁所在的基础高度。
- 也可以理解为 X 轴滑台参考高度。

影响：

- 它越大，整个机械臂挂载平面越高。

### `gantry_x_initial`

物理意义：

- 启动时 X 轴滑台相对于龙门架参考位置的初始偏移量。

### `gantry_y_initial`

物理意义：

- 启动时 Y 轴滑台的初始偏移量。

### `gantry_z_initial`

物理意义：

- 启动时 Z 轴滑台的初始偏移量。

注意：

- 在本模型里 Z 轴通常使用负值，这表示从顶梁往下移动。

### `gantry_x_min/max`

物理意义：

- X 轴最小和最大可达范围。

### `gantry_y_min/max`

物理意义：

- Y 轴最小和最大可达范围。

### `gantry_z_min/max`

物理意义：

- Z 轴上下运动的最小和最大范围。

## 为什么龙门架三轴能改变机械臂基座位置

因为机械臂基座 `ur10` 是沿着这条链挂下来的：

```text
world
 -> gantry_structure_link
 -> gantry_x_carriage_link
 -> gantry_y_carriage_link
 -> gantry_z_carriage_link
 -> ur_mount_link
 -> ur10
```

只要链路前面的某一段位置变了，后面的所有子节点都会跟着变。

所以：

- 改 `gantry_x_joint`，机械臂基座整体沿 X 平移。
- 改 `gantry_y_joint`，机械臂基座整体沿 Y 平移。
- 改 `gantry_z_joint`，机械臂基座整体沿 Z 平移。

这不是只移动某个关节，而是移动整台机械臂的“安装根部”。

## 这和直接动机械臂关节有什么本质区别

区别非常大：

- 动机械臂关节
  - 改变的是机械臂自身姿态。
  - 机械臂基座不动。
- 动龙门架关节
  - 改变的是机械臂基座在世界坐标中的位置。
  - 整台机械臂的工作空间整体发生平移。

所以龙门架的价值是：

让机械臂可以把自己的“作业原点”搬到别的位置。

## 这次为什么还改了 Gazebo visual

原先机械臂本体依赖 DAE mesh 做 visual。

在 RViz2 中这通常问题不大，但在 Gazebo Classic 中，某些 mesh 资源可能出现以下问题：

- 不显示
- 材质异常
- 显示不稳定

所以这次为 Gazebo 增加了兼容 visual：

- 不改 link
- 不改 joint
- 不改 collision
- 只在 visual 上为 Gazebo 提供稳定的圆柱几何体表示

这样既保证“看得见”，又不影响控制和规划。

## 如果你只想抓住最核心的一句话

龙门架能改变机械臂基座位置，不是因为它“控制了机械臂”，而是因为机械臂的基座本来就被挂在龙门架三轴运动链的末端。
