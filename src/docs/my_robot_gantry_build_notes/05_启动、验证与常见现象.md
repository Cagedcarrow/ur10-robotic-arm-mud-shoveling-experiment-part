# 启动、验证与常见现象

## 这篇文档解决什么问题

这篇文档回答三个实际问题：

- 我应该怎么启动这套系统。
- 我怎么确认它真的运行正常。
- 启动和退出过程中出现哪些现象是已知的、可以接受的。

## 启动前准备

每次开新终端，先执行：

```bash
source /opt/ros/humble/setup.bash
source /root/ur10_ws/install/setup.bash
```

为什么这一步必须做：

- 第一行加载 ROS 2 Humble 环境。
- 第二行加载你工作空间里已经构建好的包、启动文件和 Python 可执行程序。

如果漏掉这一步，最常见的现象是：

- 找不到包
- 找不到 launch 文件
- 找不到 Python 节点

## 最常用的完整启动命令

```bash
ros2 launch my_robot sim_planning_gantry.launch.py gui:=true start_rviz:=true
```

参数含义：

- `gui:=true`
  - 打开 Gazebo 图形界面。
- `start_rviz:=true`
  - 启动 RViz2。

这条命令会触发完整链路：

- Gazebo 启动
- `robot_state_publisher` 启动
- 机器人生成到 Gazebo
- 控制器加载
- 龙门架初始化
- MoveIt 启动
- RViz2 启动

## 如果你只想看仿真，不想开 RViz2

可以这样启动：

```bash
ros2 launch my_robot sim_planning_gantry.launch.py gui:=true start_rviz:=false
```

这适合：

- 只排查 Gazebo 模型显示
- 只排查控制器加载

## 如果你只想在无界面模式验证启动链路

可以这样启动：

```bash
ros2 launch my_robot sim_planning_gantry.launch.py gui:=false start_rviz:=false
```

这适合：

- 命令行验证
- 自动化测试
- 远程环境

## 建议的验证顺序

下面是一套从基础到完整的验证方法。

### 1. 验证 Xacro 展开

```bash
xacro /root/ur10_ws/src/my_robot_xacro/my_robot/urdf/my_robot_gantry.urdf.xacro > /tmp/my_robot_gantry.urdf
```

你应该看到：

- 命令成功结束
- `/tmp/my_robot_gantry.urdf` 被生成

这一步说明：

- 组合模型的 Xacro 语法没有问题
- 参数替换可以正常完成

### 2. 验证 URDF 结构

```bash
check_urdf /tmp/my_robot_gantry.urdf
```

你应该看到：

- `Successfully Parsed XML`

这一步说明：

- 最终生成的 URDF 结构合法
- 关节树能被正常解析

### 3. 启动仿真并观察模型

```bash
ros2 launch my_robot sim_planning_gantry.launch.py gui:=true start_rviz:=false
```

你应该观察到：

- Gazebo 中能看到龙门架
- Gazebo 中能看到真实 UR10 外形本体
- 机械臂挂在龙门架下方，而不是漂浮在奇怪位置

说明：

当前实现中：

- Gazebo 使用高保真 STL visual mesh
- RViz2 使用原始 DAE mesh

两边看到的都应当是真实机械臂外形，而不是圆柱拼接的简化模型。

### 4. 验证控制器加载

启动后，重点关注以下控制器是否成功激活：

- `joint_state_broadcaster`
- `joint_trajectory_controller`
- `gantry_trajectory_controller`

如果你看到它们加载并激活成功，说明：

- Gazebo 与 `ros2_control` 的接口是通的
- 机械臂和龙门架的控制分工生效了

### 5. 验证龙门架初始化动作

在启动链路的最后，系统会调用龙门架初始化脚本，把三轴移动到 `gantry_config.json` 中的初始位置。

如果成功，你通常会看到：

- 启动后过一会儿龙门架移动到指定位置
- 日志里提示动作执行成功

这一步说明：

- `gantry_trajectory_controller` 正常
- action 接口可用

### 6. 验证 MoveIt 与 RViz2 联动

用完整启动命令：

```bash
ros2 launch my_robot sim_planning_gantry.launch.py gui:=true start_rviz:=true
```

你应该看到：

- RViz2 打开
- MotionPlanning 面板可用
- 机器人模型和场景正常显示

这一步说明：

- MoveIt 启动成功
- RViz2 成功接入 `robot_description` 和规划场景

### 7. 验证规划与执行

你可以运行：

```bash
ros2 run ur10_examples_py moveit_py_demo
```

如果你的调用方式里有额外参数，就按你当前项目实际使用的方式传入。

理想现象：

- MoveIt 规划成功
- 执行结果显示 `SUCCEEDED`
- Gazebo 控制器侧出现类似 `Goal reached, success!` 的成功反馈

这一步说明：

- MoveIt 到控制器的执行链路是通的
- 仿真里的机械臂关节真的跟着动了

## `moveit_py_demo.py` 在当前项目中的作用

这个脚本的意义不只是“一个示例”。

在当前项目里，它实际承担了三个角色：

- 验证 MoveIt 规划是否可用
- 验证控制器执行是否可用
- 验证这次通用化改造后，脚本能复用到 `my_robot`

所以每次大改完之后，用它做一次完整验证是非常有价值的。

## 为什么现在它可以复用到 `my_robot`

因为这次你把整套系统做成了“接口一致”的形式：

- 机器人描述统一
- MoveIt 配置统一
- 控制器映射统一
- 规划组明确为 `my_robot_manipulator`
- 末端 link 明确为 `shovel_tip`

这让原本更偏示例性质的 `moveit_py_demo.py`，能够变成一个真正可复用的测试入口。

## 已知现象 1：`gazebo_ros2_control` 等待 `robot_state_publisher` 偏慢

现象描述：

- 启动时 Gazebo 和控制器链路之间可能会有一段明显等待。
- 看起来像“控制器怎么还没起来”。

原因理解：

- `gazebo_ros2_control` 需要从 `robot_state_publisher` 所在节点获取 `robot_description`。
- 启动时序稍慢时，这里就会出现等待。

如何判断这是不是正常现象：

- 只要后续控制器能正常激活
- 龙门架初始化能执行
- 轨迹能成功跑通

那这类等待就属于已知且可接受现象。

## 已知现象 2：`move_group` 在 Ctrl-C 退出时偶发 segfault

现象描述：

- 你按 `Ctrl-C` 结束系统时，`move_group` 可能会在退出阶段报一次性段错误。

为什么目前可以接受：

- 它发生在退出阶段，不是在运行阶段。
- 它不会影响：
  - Gazebo 里模型生成
  - 控制器激活
  - MoveIt 规划
  - 轨迹执行成功

因此当前可以把它归类为：

已知退出时异常，不影响核心功能验证。

## 如果启动后还是看不到真实 UR10 mesh，优先排查什么

建议按这个顺序查：

1. 是否真的 `source` 了正确的工作空间。
2. 是否已经重新构建并安装了 `my_robot` 包。
3. 是否使用的是当前更新后的 `my_robot_gantry.urdf.xacro`。
4. `package://my_robot/meshes/...` 对应的 DAE/STL mesh 是否已经安装到工作空间的 `install` 目录。
5. Gazebo 中是否已经成功生成 `my_robot_gantry` 实体。
6. 是否因为旧进程残留导致“实体已存在”。
7. 是否启动的是图形界面 `gui:=true`。

特别提醒：

如果上一次 Gazebo 没关干净，重新启动时可能出现：

- `Entity [my_robot_gantry] already exists`
- 控制器 already loaded

这不是本次模型问题，而是旧进程残留问题。

另外，如果 Gazebo 里再次出现“只有龙门架、没有真实机械臂 mesh”，优先排查 STL/DAE 资源解析问题和 Gazebo Classic 对 mesh 格式的兼容，而不是回退到圆柱兼容模型。

## 一个推荐的日常检查习惯

每次你改完模型或控制链路后，建议至少做下面三步：

1. `xacro` 展开
2. `check_urdf`
3. 启动 `sim_planning_gantry.launch.py`

如果这三步都过了，再继续做：

4. 控制器验证
5. `moveit_py_demo.py` 轨迹执行验证

这个习惯能帮你很快判断问题到底出在：

- 模型描述层
- 仿真层
- 控制层
- 规划层

## 如果你只记住一句话

对这套系统来说，真正的“验证完成”不是只看到模型出现，而是 Gazebo、控制器、MoveIt、RViz2 和轨迹执行整条链都跑通。
