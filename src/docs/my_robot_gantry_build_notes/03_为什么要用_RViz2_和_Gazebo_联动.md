# 为什么要用 RViz2 和 Gazebo 联动

## 这篇文档解决什么问题

很多刚学 ROS 2 的同学会问：

- 我已经有 Gazebo 了，为什么还要 RViz2？
- 我已经能在 RViz2 里看机器人了，为什么还要 Gazebo？

这篇文档就是专门回答这个问题的。

## 先给结论

在本项目里：

- Gazebo 负责“仿真世界和执行”
- RViz2 负责“可视化和交互”
- MoveIt 负责“规划”

三者不是替代关系，而是分工合作关系。

## Gazebo 负责什么

Gazebo 是一个仿真器。

在本项目里，它主要负责：

- 把机器人实体真正生成到仿真世界中。
- 运行 `gazebo_ros2_control`。
- 让控制器去驱动仿真关节。
- 执行动作后反馈关节状态。

你可以把 Gazebo 理解成：

“机器人真的在一个虚拟世界里动起来的地方。”

## RViz2 负责什么

RViz2 是一个可视化工具。

在本项目里，它主要负责：

- 显示 TF 坐标树结果。
- 显示机器人当前状态。
- 显示 MoveIt 的规划轨迹。
- 提供 MotionPlanning 面板。
- 显示龙门架的交互 Marker。

你可以把 RViz2 理解成：

“人类观察和操作机器人状态的窗口。”

## MoveIt 负责什么

MoveIt 不是仿真器，也不是单纯的显示工具。

它负责：

- 根据目标位姿或目标关节值做路径规划。
- 检查碰撞和约束。
- 生成机械臂轨迹。
- 把轨迹发送给执行控制器。

所以：

- Gazebo 不是规划器。
- RViz2 不是规划器。
- MoveIt 才是规划器。

## 为什么单用 Gazebo 不够

如果只有 Gazebo，你可以看到模型在仿真世界里出现，也可以看到控制器执行动作。

但你会缺很多东西：

- 不方便查看 MoveIt 的规划结果。
- 不方便直接用 MotionPlanning 面板调试。
- 不方便直观看到规划前、规划后、执行中的差异。
- 不方便使用交互 Marker。

对于调试规划系统来说，这会非常痛苦。

## 为什么单用 RViz2 不够

如果只有 RViz2 和 MoveIt，你可以：

- 看机器人模型
- 设目标
- 做轨迹规划

但你缺少真正的仿真执行环境：

- 看不到机器人在 Gazebo 世界里是否真的按控制器执行
- 验证不了 `gazebo_ros2_control`
- 验证不了实体生成、控制器加载、仿真反馈这条链

换句话说：

只有 RViz2，你更多是在看“规划和显示是否合理”；
只有 Gazebo，你更多是在看“仿真和执行是否合理”。

而本项目需要两者同时成立。

## 本项目里二者是怎么联动起来的

联动的基础是：大家共享同一份机器人描述和同一套 joint states。

### 1. 同一份 `robot_description`

Gazebo 需要 `robot_description` 来生成实体。

MoveIt 和 RViz2 也需要 `robot_description` 来知道机器人长什么样。

这意味着：

- 大家必须基于同一份机器人结构工作。
- 不能 Gazebo 用一份模型，MoveIt 用另一份模型。

否则就会出现：

- Gazebo 里看到的机器人和 RViz2 里规划的机器人不是同一台。
- 关节名不一致。
- 末端 link 不一致。
- 规划能成功但执行不了。

### 2. 同一套关节状态

控制器在 Gazebo 中执行后，会产生关节状态。

这些关节状态会进入：

- `joint_states`
- `robot_state_publisher`
- TF
- MoveIt 当前状态监视器
- RViz2 显示

于是你在 RViz2 中看到的姿态，就能和 Gazebo 中真实执行后的姿态保持一致。

### 3. Gazebo 执行，RViz2 观察

在这套系统里，常见流程是：

1. 你在 RViz2 里给 MoveIt 一个目标。
2. MoveIt 计算轨迹。
3. MoveIt 把轨迹发给控制器。
4. 控制器在 Gazebo 中驱动仿真机器人执行。
5. Gazebo 回传关节状态。
6. RViz2 更新显示。

这就是“Gazebo 执行 + RViz2 观察 + MoveIt 规划”的典型联动链路。

## 在本项目里，RViz2 还有一个额外价值

本项目不只是机械臂关节规划，还有龙门架位置调整。

而龙门架调整这件事，如果只靠命令行改参数，体验会很差。

所以这里又引入了 RViz2 中的 Interactive Marker：

- 你拖动 Marker
- 程序把拖动位置换算成 `gantry_x/y/z_joint`
- 再通过 action 发给 `gantry_trajectory_controller`

这样你不是在“改数字”，而是在“拖一个可见的空间控制点”。

对于新手来说，这个差别非常大。

## 为什么本项目必须联动，而不是只做其中一边

因为本项目验证的不只是“模型能显示”，而是以下整条链都要成立：

- 组合机器人结构正确
- Gazebo 实体生成成功
- 控制器成功激活
- MoveIt 成功规划
- 规划轨迹能实际执行
- RViz2 能正确显示并辅助交互

只有 Gazebo 或只有 RViz2，都不能完整覆盖这条链。

## 一个很好记的类比

你可以把它们想象成：

- Gazebo：工厂车间
- MoveIt：路径规划工程师
- RViz2：监控与操作台

只开操作台，不代表工厂真的在运行。
只开工厂，不代表你看清楚了规划过程。
要做完整联调，就要三者一起工作。

## 本项目中的实际体现

在 `sim_planning_gantry.launch.py` 中，总入口做的正是这件事：

- 先包含 Gazebo 启动链路
- 再延时启动 MoveIt 链路

这么做的意义是：

- 先让仿真世界和机器人实体稳定起来
- 再让规划系统接入

这能减少启动时因为时序问题导致的异常。

## 如果你只记一句话

在本项目里，Gazebo 负责“机器人真正在仿真里动”，RViz2 负责“你看见它、理解它、操作它”，而 MoveIt 负责“决定它怎么动”。
