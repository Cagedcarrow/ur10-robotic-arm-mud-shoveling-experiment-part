# 这次实际改了什么

## 这篇文档解决什么问题

前面几篇文档主要在讲：

- 问题现象
- 原理
- 根因

这篇文档专门讲“这次源码层面实际改了什么”，但不会把内容写成纯粹的 diff 清单。重点是让你知道每个改动为什么存在、它解决的是哪一类问题。

## 改动 1：把 `铲子` 改成 `shovel_link`

### 修改前

模型里存在中文 link 名 `铲子`，并且它还出现在：

- URDF/Xacro 的 link 定义里
- 固定关节 child 里
- MoveIt 的 SRDF disable collision 配置里

### 修改后

统一改成：

```text
shovel_link
```

### 这个改动解决什么问题

它主要解决的是 Gazebo Classic 对中文命名不稳定的问题。

这次最关键的直接收益是：

- 不再触发 `invalid UTF-8 data`
- Gazebo 在处理 visual/collision 相关名称时更稳定

### 你以后该怎么记

在 ROS 2 里，不是所有地方都绝对禁止中文，但只要这个名字会进入 Gazebo Classic、MoveIt、控制器或其他工具链，优先用 ASCII。

也就是：

- 注释可以中文
- 文档可以中文
- 运行时实体名尽量不要中文

## 改动 2：把铲斗 mesh 从 `package://...` 改成 `file://...`

### 修改前

铲斗 mesh 用的是：

```text
package://my_robot/meshes/base_link.STL
```

### 修改后

改成：

```text
file://$(find my_robot)/meshes/base_link.STL
```

### 这个改动解决什么问题

它不是为了“看起来更高级”，而是为了让 Gazebo 最终拿到更稳定的真实路径。

本次问题里，Gazebo 最终尝试加载的是：

```text
model://my_robot/meshes/base_link.STL
```

它找不到，所以铲斗 mesh 显示失败。

改成 `file://...` 后，展开结果会变成绝对路径，更利于 Gazebo 直接定位文件。

### 你以后该怎么记

对于 mesh 路径：

- RViz 能解析，不代表 Gazebo 一定也稳定。
- 最终要看“Gazebo 真正吃到的 SDF URI 是什么”。

## 改动 3：把 `gantry_only.world` 从外部模型依赖改成内联 world 元素

### 修改前

world 文件依赖：

- `model://ground_plane`
- `model://sun`

### 修改后

直接把：

- 地面 plane
- directional light

写进了 world 文件本体。

### 这个改动解决什么问题

它主要解决的是：

- Gazebo 启动时等待模型数据库
- 启动时间过长
- 启动过程不稳定

### 你以后该怎么记

如果你的 world 非常简单，只是需要：

- 一个地面
- 一个光源

那直接写进 world 往往比依赖外部模型更稳。

## 改动 4：把 UR10 和 gantry 的 `ros2_control` 合并成一个 system

### 修改前

机械臂和龙门架各自有一套 `ros2_control` 定义。

这会让 `gazebo_ros2_control` 在一个模型里处理两套 system。

### 修改后

在整机 Xacro 里统一成一套 `ros2_control`：

- 同一个 system 管理 UR10 关节
- 同一个 system 管理 gantry 三个关节

同时把机械臂宏里原本自动注入的 `ros2_control` 能力改成可选开关，让整机组合时避免重复声明。

### 这个改动解决什么问题

它直接消掉了这条错误：

```text
Parameter 'hold_joints' has already been declared
```

并让 controller manager 的行为更清晰：

- 一套整机控制接口
- 一个 controller manager
- 三个控制器在这套 manager 下正常加载

### 你以后该怎么记

如果你在 Gazebo 中把多个子系统拼成一个整体机器人，尽量提前想清楚：

- 是每个子模块单独带一套控制 system
- 还是整机统一成一套 system

像这次这种“组合机器人作为一个实体一起 spawn 到 Gazebo”的场景，统一成一套 system 更稳。

## 改动 5：同步更新 SRDF 的 link 名

### 修改前

MoveIt 的 SRDF 里还引用旧的 `铲子`。

### 修改后

同步改成 `shovel_link`。

### 这个改动解决什么问题

它的作用不是修 Gazebo 显示本身，而是防止：

- URDF 和 SRDF 命名不一致
- MoveIt 继续引用旧 link 名
- 后续出现碰撞配置或规划链路错误

## 实际涉及的关键位置

这次最关键的改动集中在这几类文件：

- `my_robot` 的 URDF/Xacro
- `my_robot_moveit_config` 的 SRDF
- `ur10_perception` 的 world 文件

你以后回看时，只要先想到这三层，就能比较快地定位：

- 显示问题多半在 URDF/Xacro 或 world
- 控制器问题多半在 `ros2_control`
- 规划一致性问题多半还要看 SRDF

## 最后用一句话概括这次修改

这次不是“换了一个启动命令就好了”，而是把同一条仿真链路上的 4 类问题一起收拾干净了：

- 名称问题
- mesh 路径问题
- world 启动问题
- 控制系统重复声明问题
