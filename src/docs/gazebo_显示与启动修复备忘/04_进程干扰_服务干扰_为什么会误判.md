# 进程干扰、服务干扰，为什么会误判

## 这篇文档解决什么问题

这篇文档专门讲一个非常容易坑新手、但又特别真实的问题：

为什么你明明已经改对了代码，日志却还像没改一样？

答案往往不是“你又改错了”，而是：

- 旧进程没退出
- 旧服务还在
- 新旧 launch 混在一起了

这类问题如果不理解，会让你在排障时不停怀疑自己，甚至沿着错误方向继续改代码。

## 先理解几个关键进程和服务

### `gzserver`

它是 Gazebo 的仿真服务器进程。

你可以把它理解成：

- 世界和模型真正活着的地方

### `robot_state_publisher`

它会读取 `robot_description`，把机器人关节树发布成 TF。

### `spawn_entity.py`

它会把 `robot_description` 送给 Gazebo 的 `/spawn_entity` 服务，要求 Gazebo 生成机器人实体。

### `controller_manager`

它负责管理控制器的加载、配置和激活。

### `ros2 daemon`

它不是仿真本体，但会缓存 ROS 图信息。大多数时候没问题，排障时偶尔会让你看到延迟或旧信息。

## 什么叫“进程干扰”

举个简单例子。

你第一次启动：

```bash
ros2 launch my_robot sim_planning_gantry.launch.py gui:=false start_rviz:=false
```

中途没完全退出干净，`gzserver`、`robot_state_publisher`、`spawn_entity.py` 还留着。

然后你改了代码，再启动第二次：

```bash
ros2 launch my_robot sim_planning_gantry.launch.py gui:=false start_rviz:=false
```

这时候就可能出现：

- 新的 `robot_state_publisher`
- 旧的 `gzserver`
- 旧的 `/spawn_entity`
- 新的 `spawn_entity.py`

互相混在一个 ROS 图里。

这就叫进程干扰。

## 什么叫“服务干扰”

ROS 里很多节点是靠服务通信的。

例如：

- `/spawn_entity`
- `/controller_manager/list_controllers`

如果旧进程还活着，它就还可能继续提供旧服务。

于是你在第二次启动时，新的节点可能会连到旧服务上。

这样就会出现非常诡异的现象：

- 你以为自己在和“新代码”交互
- 实际上节点连上的却是“旧 Gazebo 或旧 controller manager”

## 为什么这会让日志看起来前后矛盾

比如这次排障里，源码已经改成：

- `shovel_link`
- `file://.../base_link.STL`

但日志里却还出现：

- `铲子`
- `model://my_robot/meshes/base_link.STL`

如果你不知道有残留进程，就很容易得出错误结论：

- “是不是安装空间没更新？”
- “是不是 xacro 根本没读到我改的文件？”
- “是不是 launch 引用了别的包？”

这些方向不一定错，但在那一刻，更大的可能其实是：

- 你看到的是旧进程打印的日志
- 或者新节点连到了旧服务

## 这次修复里，残留进程具体怎么干扰了判断

这次排障过程中，确实发现过多轮旧进程还在：

- 旧 `gzserver`
- 旧 `robot_state_publisher`
- 旧 `spawn_entity.py`
- 旧 `move_group`

它们的后果是：

1. 新的 launch 不一定连接到新的 Gazebo。
2. `/spawn_entity` 可能来自旧的 Gazebo。
3. `robot_description` 的观察结果可能不是你以为的那一份。
4. 你看到的报错不一定对应当前这轮启动。

## 为什么“先清进程再验证”这么重要

因为你在修的是“系统行为”，不是单个文件。

对于这种多进程系统，如果不先保证环境干净，任何验证结果都可能被污染。

这就像你在调试一个 Web 服务，但旧版本服务还在后台监听旧端口。你再怎么改代码，观察到的结果也不可信。

## 安全的排障顺序

注意，这里重点是“顺序”，不是“背命令”。

### 第一步：先看有没有旧进程

先确认：

- 有没有旧 `gzserver`
- 有没有旧 `robot_state_publisher`
- 有没有旧 `spawn_entity.py`
- 有没有旧 `move_group`

如果有，就先把它们处理干净。

### 第二步：必要时停掉 ROS daemon

这一步不是每次都必须，但在你觉得 ROS 图显示不对、服务列表怪怪的时候，值得做一下。

### 第三步：重新 source 环境

```bash
source /opt/ros/humble/setup.bash
source /root/ur10_ws/install/setup.bash
```

### 第四步：从无 GUI 启动开始验证

```bash
ros2 launch my_robot sim_planning_gantry.launch.py gui:=false start_rviz:=false
```

因为无 GUI 模式下，你更容易专注看日志和服务。

### 第五步：确认核心链路稳定后，再开 GUI

核心链路包括：

- `spawn_entity` 成功
- `controller_manager` 可用
- 控制器能激活

确认这些没问题后，再去开 `gui:=true start_rviz:=true`。

## 新手最容易犯的误区

### 误区 1：一看到报错就改源码

有些报错确实要改源码，但有些报错只是旧进程留下的历史输出。

### 误区 2：只看一条日志，不看整条链路

如果只盯着一条报错，你很可能错过：

- 这条报错来自旧进程
- 这条报错其实是前一个问题的连锁反应

### 误区 3：把“服务存在”当成“服务可正常工作”

比如 `/controller_manager/list_controllers` 出现在服务列表里，不一定代表它已经稳定可用。

还要看：

- 对应节点是否真的活着
- 服务是否能响应
- Gazebo 插件是否已完整初始化

## 以后怎么避免再次被误导

给自己定一个固定习惯：

每次做 Gazebo/MoveIt 这类系统级排障前，先做三件事：

1. 检查残留进程
2. 确认环境 source 正确
3. 从无 GUI 启动开始验证

这 3 步看起来“有点麻烦”，但它能帮你省掉很多无意义的来回猜测。
