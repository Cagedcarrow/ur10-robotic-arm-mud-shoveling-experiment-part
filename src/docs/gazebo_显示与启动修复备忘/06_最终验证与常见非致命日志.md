## 两类目前可以接受的非致命日志

下面这两类日志，在当前状态下不影响“Gazebo 正常启动、模型正常显示、控制器正常工作”。

## 1. Octomap / 3D sensor plugin 相关日志

常见形式：

```text
No 3D sensor plugin(s) defined for octomap updates
```

### 它是什么意思

这是 MoveIt 在说：

- 目前没有给 octomap 配置 3D 传感器更新插件

### 为什么现在不用把它当致命问题

因为本次修复目标是：

- Gazebo 显示
- 启动速度
- 控制器链路

并不是“立刻把深度相机接入 MoveIt 的 Octomap 障碍物更新”。

也就是说，它提醒的是“环境感知功能还没完全接上”，不是“仿真启动失败”。

## 2. `/recognize_objects not available`

常见形式：

```text
Action server: /recognize_objects not available
```

### 它是什么意思

这是 RViz/MoveIt 某些可视化或感知相关功能在提示：

- 当前没有对应的对象识别 action server

### 为什么现在可以接受

因为它不影响：

- Gazebo 启动
- 机械臂显示
- 控制器加载
- 轨迹规划基础链路

所以它属于“功能未接全”的提示，不是“当前系统坏了”。

## 什么才算真正的致命问题

以后你排障时，可以这样区分。

### 致命问题

会直接阻止主流程工作的，比如：

- Gazebo 起不来
- 机器人实体 spawn 失败
- mesh 导致模型显示失败
- `controller_manager` 起不来
- 控制器无法激活

### 非致命告警

不会阻止主流程，但说明某些高级功能没接好，比如：

- octomap 更新插件没配置
- 对象识别 action server 不存在
- 某些控制器参数将来会废弃

## 最后给新手的一句话

系统“修好”不等于“日志一条红字都没有”。

更实际的判断标准是：

- 你关心的主流程是否已经稳定可用
