# 新手排查流程

## 这篇文档解决什么问题

如果你以后再次遇到下面这些情况：

- Gazebo 启动很慢
- 机械臂不显示
- 机械臂只显示一部分
- `spawn_entity` 卡住
- 控制器起不来
- 日志看起来很乱

那就不要从头乱猜。直接按这篇文档的顺序查。

## 先记住一个原则

排障时不要一上来就开全套图形界面。

优先用：

```bash
source /opt/ros/humble/setup.bash
source /root/ur10_ws/install/setup.bash
ros2 launch my_robot sim_planning_gantry.launch.py gui:=false start_rviz:=false
```

原因很简单：

- 控制台更干净
- 日志更容易看
- 不会被 GUI 渲染和窗口问题分散注意力

## 第 1 步：先确认是不是残留进程

### 你要看什么

先看有没有旧的：

- `gzserver`
- `gzclient`
- `robot_state_publisher`
- `spawn_entity.py`
- `move_group`

### 正常现象

- 只有你当前这轮启动对应的进程

### 异常现象

- 发现多轮旧进程还活着
- 明明没在运行，但相关进程还在后台

### 如果异常，说明什么

优先怀疑是旧进程干扰，而不是源码又错了。

## 第 2 步：看 `robot_state_publisher` 输出的 link 名

### 你要看什么

启动时，`robot_state_publisher` 会打印它收到的 segment。

### 正常现象

应该能看到：

```text
got segment shovel_link
```

### 异常现象

如果你还看到旧名字，例如：

```text
got segment 铲子
```

那就说明：

- 当前启动链路可能还在吃旧模型
- 或者你连接到的不是你以为的那一轮环境

## 第 3 步：看 `spawn_entity` 是否成功

### 你要看什么

重点找这类输出：

```text
Spawn status: SpawnEntity: Successfully spawned entity [my_robot_gantry]
```

### 正常现象

- `spawn_entity` 成功
- 机器人实体真正进了 Gazebo

### 异常现象

- 一直等 `/spawn_entity`
- 调用失败
- 报实体生成失败

### 如果异常，说明什么

优先怀疑：

- Gazebo 没真正起来
- world 有问题
- 模型解析有问题

## 第 4 步：看 Gazebo 有没有 mesh 相关错误

### 你要看什么

重点找：

```text
Failed to find mesh file ...
No mesh specified
```

### 正常现象

- 没有 mesh 找不到的错误

### 异常现象

- 某个 STL/DAE 找不到
- `model://...` 被解析失败

### 如果异常，说明什么

优先怀疑：

- mesh URI 写法不适合 Gazebo
- 安装路径或展开结果不对

## 第 5 步：看有没有 world 卡住

### 你要看什么

重点找：

```text
Waiting for model database update to complete...
```

### 正常现象

- world 很快完成加载

### 异常现象

- Gazebo 长时间停在 world 相关日志

### 如果异常，说明什么

优先怀疑不是机器人本体，而是 world 本身在等外部模型数据库。

## 第 6 步：再看控制器链路

### 你要看什么

重点看：

- `/controller_manager/list_controllers` 是否可用
- `joint_state_broadcaster` 是否加载激活
- `joint_trajectory_controller` 是否加载激活
- `gantry_trajectory_controller` 是否加载激活

### 正常现象

应当能看到类似：

```text
Configured and activated joint_state_broadcaster
Configured and activated joint_trajectory_controller
Configured and activated gantry_trajectory_controller
```

### 异常现象

- 一直等 `/controller_manager/list_controllers`
- `controller_manager` 没响应
- 控制器只 load 但没 activate

### 如果异常，说明什么

优先怀疑两类问题：

1. Gazebo 前面还没真正稳定
2. `ros2_control` 声明方式有问题

## 第 7 步：最后再开 GUI 验证

当你已经确认下面这些都正常之后，再开 GUI：

- `spawn_entity` 成功
- mesh 不报错
- world 不再卡住
- 三个控制器能激活

然后再执行：

```bash
ros2 launch my_robot sim_planning_gantry.launch.py gui:=true start_rviz:=true
```

## 一份适合新手的快速判断表

### 现象：Gazebo 很慢

先怀疑：

- world 卡模型数据库
- 旧 Gazebo 进程没退出

### 现象：机械臂不显示

先怀疑：

- UTF-8 命名问题
- mesh URI 问题
- `spawn_entity` 没成功

### 现象：`controller_manager` 起不来

先怀疑：

- 前面的 Gazebo/model/plugin 还没稳定
- `ros2_control` 重复或冲突

### 现象：日志前后矛盾

先怀疑：

- 旧进程
- 旧服务
- ROS 图缓存

## 对新手最重要的一句话

不要把 Gazebo 问题直接理解成“渲染问题”。

更多时候，它其实是：

- 模型描述问题
- 路径解析问题
- world 依赖问题
- 控制插件问题
- 或者环境里还残留旧进程的问题

只要你按顺序查，很多问题会比想象中更容易定位。
