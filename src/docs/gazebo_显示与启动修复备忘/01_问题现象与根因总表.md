# 问题现象与根因总表

## 这篇文档解决什么问题

这篇文档把这次修复中遇到的主要问题，按“现象 -> 关键日志 -> 根因 -> 修复方式”的形式整理出来。

你以后如果再次遇到类似情况，可以先来这里对照，不用从头猜。

## 总表

| 现象 | 常见关键日志 | 真正根因 | 最终修复 |
| --- | --- | --- | --- |
| Gazebo 启动十几分钟都没好，机械臂也不显示 | `String field 'gazebo.msgs.Visual.name' contains invalid UTF-8 data` | 模型里存在中文 link 名 `铲子`，Gazebo Classic 对这类名称不稳定 | 把 `铲子` 统一改成 ASCII 名称 `shovel_link` |
| Gazebo 启动了，但铲斗或者整机显示不完整 | `Failed to find mesh file [model://my_robot/meshes/base_link.STL]` | 铲斗 mesh 的 URI 经过 URDF -> SDF 转换后，变成 Gazebo 找不到的 `model://...` | 把铲斗 mesh URI 改成稳定的 `file://$(find my_robot)/meshes/base_link.STL` |
| Gazebo 看起来“卡住”，控制器一直起不来 | `Waiting for model database update to complete...` | world 文件里有 `model://ground_plane` 和 `model://sun`，Gazebo 在等待模型数据库 | 在 world 里内联地面和光源，去掉对模型数据库的依赖 |

| 明明已经改了代码，但日志还像旧问题没修 | 新日志和旧日志混在一起，甚至看到已经不该存在的旧 URI | 旧 `gzserver`、旧 `spawn_entity.py`、旧 `robot_state_publisher` 没退出，服务和主题串线 | 排障前先清理残留进程，必要时停止 `ros2 daemon`，再重新验证 |

## 每个问题再展开解释一遍

### 1. Gazebo 卡住且机械臂不显示

这是最容易让人误以为“mesh 太大”或者“Gazebo 性能太差”的现象。

但这次真正抓到的关键日志是：

```text
String field 'gazebo.msgs.Visual.name' contains invalid UTF-8 data
String field 'gazebo.msgs.Collision.name' contains invalid UTF-8 data
```

这说明 Gazebo 在处理模型内部名字时就出问题了，根因不是 STL 文件太大，而是名字本身不适合 Gazebo Classic。

### 2. 铲斗 mesh 找不到

这类问题的典型日志是：

```text
Failed to find mesh file [model://my_robot/meshes/base_link.STL]
```

新手最容易误解成：

- “这个 STL 文件不存在”
- “是不是 install 没有拷进去”

其实这次文件本身是存在的，问题在于 Gazebo 最终拿到的 URI 形式不对。

### 3. `controller_manager` 看起来像没起来

你可能会看到：

```text
waiting for service /controller_manager/list_controllers
```

这时候不要马上只盯着控制器 YAML 看。因为如果 Gazebo 前面就被 mesh 或 world 卡住了，控制插件根本没机会完整起来。

所以排障顺序很重要：先确认显示和 world，再确认控制器。

### 4. world 卡模型数据库

这次 Gazebo 日志里出现过：

```text
Waiting for model database update to complete...
```

它的含义不是“机器人模型有错”，而是 Gazebo 的世界文件还在等待外部模型数据库这一步完成。

如果你不理解这一层，就很容易以为“Gazebo 服务器卡死了”。

### 5. 旧进程让日志看起来像穿越

这是最坑人也最容易忽略的问题。

你可能已经把源码改成：

- `shovel_link`
- `file://.../base_link.STL`

但日志里却还出现：

- `铲子`
- `model://my_robot/meshes/base_link.STL`

这时候要高度怀疑：你看到的不是“新代码没生效”，而是“旧进程还没退出”。

## 以后遇到类似问题的使用建议

如果你现在又遇到 Gazebo 显示异常，建议按下面顺序去想：

1. 这是名字问题吗。
2. 这是 mesh URI 问题吗。
3. 这是 world 启动问题吗。
4. 这是控制器问题吗。
5. 这是残留进程导致的假象吗。

按这个顺序查，通常会比直接盯着某一条报错更快。
