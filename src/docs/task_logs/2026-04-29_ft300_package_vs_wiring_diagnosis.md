# 本次任务总结

## 1. 任务目标
继续实测，判断 FT300 当前无数据问题更偏向“ROS 功能包不匹配”还是“连接线/链路问题”。

## 2. 本轮新增实测

### 2.1 设备类型与驱动识别
- `lsusb`:
  - `0403:6015 FTDI`
- `udevadm info -a -n /dev/ttyUSB0`:
  - `product == "USB TO RS-485"`
  - `interface == "USB TO RS-485"`
  - 驱动 `ftdi_sio`

结论：系统把它识别为 RS-485 转换设备，不是普通 tty 假设备。

### 2.2 链路稳定性
`dmesg` 持续出现大量：
- `vhci_hcd: urb->status -104`
- 高频 `unlink->seqnum`

结论：WSL2 USBIP 虚拟 USB 链路存在明显不稳定/中断迹象。

### 2.3 协议探测（比 GUI 更底层）
我做了两类主动探测，均直接走 `/dev/ttyUSB0`，与 GUI 无关：

1. **FC16 启动流命令**（与当前包一致）
   - `09 10 01 9a 00 01 02 02 00 cd ca`
   - 波特率：`9600/19200/38400/57600/115200`
   - 结果：所有波特率均 `0` 字节回包。

2. **FC03 读寄存器扫描**（扩大覆盖）
   - 从站地址：`1~15`
   - 寄存器：`0x0000, 0x0001, 0x0100, 0x019A`
   - 波特率：`9600/19200/38400/57600/115200`
   - 结果：所有组合均 `no_response_any_slave`（无任何返回字节）。

### 2.4 ROS2 节点交叉验证
- 运行 `ft300_gui_node`（`debug_raw=true`）后，仍为“未收到有效帧”。
- 这与底层串口探测一致，说明不是 GUI 特有问题。

## 3. 结论：功能包 vs 连接线

### 结论判定
当前证据显示：
- **更像连接链路/硬件侧问题**（USBIP 稳定性或 RS-485 电气链路/接线），
- **不是功能包不匹配导致的主因**。

### 判定依据
1. 功能包逻辑已被绕过（直接串口发 Modbus），仍 0 回包。
2. 扫了多波特率 + 多从站地址 + 多寄存器，仍 0 回包。
3. 内核层出现大量 `urb->status -104`，是链路异常强证据。

## 4. 可能根因排序
1. **WSL2 USBIP 链路不稳定**（高概率）
2. **RS-485 物理链路问题**：A/B 线序、GND、终端/偏置、电源或半双工方向（高概率）
3. **设备地址或模式被改**（中概率）
4. **纯软件包不匹配**（低概率）

## 5. 下一步建议（最有效）
1. 在 Windows 侧 `usbipd detach/attach` 后立刻复测，并观察 `dmesg` 是否仍刷 `-104`。
2. 硬件侧逐项确认：
   - RS485 A/B 是否接反
   - GND 是否共地
   - 传感器供电是否稳定
   - 转换器是否真支持自动收发方向或需要配置
3. 若可行，优先用 **原生 Linux/Windows 串口工具** 直接测 Modbus 回包，先排除 WSL2 USBIP 干扰。
